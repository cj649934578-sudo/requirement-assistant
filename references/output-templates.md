# 输出模板

当用户要求正式输出，或需要统一结构与风格时，使用以下模板。

## 1. 识别摘要

```markdown
## 识别摘要
- 已识别页面:
- 已识别模块:
- 已识别字段:
- 已识别动作:
- 已识别规则:
- 已识别状态:
- 关键假设:
```

## 2. 结构化需求模型

```markdown
## 结构化需求模型
### 页面
- 页面名称:
  - 目标:
  - 入口:
  - 关键模块:

### 字段
- 字段名称:
  - 类型:
  - 是否必填:
  - 校验规则:
  - 可见性:
  - 默认值:

### 动作
- 动作名称:
  - 触发方式:
  - 前置条件:
  - 成功结果:
  - 失败结果:

### 规则
- 规则:
  - 条件:
  - 影响:

### 状态
- 状态:
  - 进入条件:
  - 退出条件:
  - 可执行动作:
```

## 3. 问题清单

```markdown
## 问题清单
### 高优先级
- [类型] 标题
  - 影响原因:
  - 依据:
  - 建议:

### 中优先级
...

### 低优先级
...
```

## 4. 待确认项

```markdown
## 待确认项
### 必须确认项
1. 问题:
   - 必须确认原因:
   - 建议选项:
     - 选项 A
     - 选项 B

### 建议优化项
1. 问题:
   - 建议先确认原因:
   - 建议选项:
     - 选项 A
     - 选项 B

### 可后续确认项
1. 问题:
   - 可延后确认原因:
   - 建议选项:
     - 选项 A
     - 选项 B
```

## 5. PRD 草稿

```markdown
# PRD 草稿

## 背景与目标

## 用户与角色

## 范围

## 页面/流程说明

## 核心交互

## 业务规则

## 异常与边界场景

## 待确认项
```

## 6. 功能清单

```markdown
## 功能清单
| 模块 | 功能点 | 触发方式 | 前置条件 | 结果 | 异常/备注 |
|---|---|---|---|---|---|
```

## 7. 测试点

```markdown
## 测试点
### 对象
- 正常路径:
- 异常路径:
- 边界场景:
- 权限场景:
- 状态/流转场景:
```

## 8. 测试用例草稿

```markdown
## 测试用例草稿
### 用例 1: [功能模块]-[测试场景标题]
- 前置条件:
- 步骤:
  1. [尽量详细描述用户操作、输入值、前置状态和触发动作]
  2. [继续补充关键步骤，不要只写“执行功能”]
- 预期结果:
- 优先级:
```

规则：
- 测试用例标题应以功能模块作为前缀。
- 示例标题：`登录模块-手机号登录成功`
- 步骤应尽量详细，能够让测试人员直接执行。
- 前置条件应在适用时明确账号状态、权限、测试数据和页面进入条件。
- 预期结果应描述用户可见结果、状态变化以及错误反馈。


## Plan-first 说明

对于以截图为主或信息不足的任务，优先先产出 `intent-summary.md`、`execution-plan.md` 和 `run-status.md`，不要直接进入完整 PRD。

待确认项格式规则：
- 不要把“一个问题 + 它的选项”压平成连续的单层编号列表。
- 只对问题本身编号。
- 选项必须作为对应问题下的缩进无序列表展示。
